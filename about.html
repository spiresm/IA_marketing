1. Introduction

Le projet "Espace IA Marketing" est une application web statique / Single Page Application (SPA) riche en fonctionnalités. Elle utilise intensivement JavaScript côté client et s'appuie sur les Netlify Functions pour ses interactions serveur, notamment la persistance des données via l'API GitHub et l'intégration avec d'autres services tiers.

2. Architecture Générale

Le site est structuré comme une collection de pages HTML statiques, dynamisées par du JavaScript côté client. Le "backend" est entièrement découplé et implémenté via des **Netlify Functions** (fonctions serverless) qui interagissent avec diverses APIs externes (GitHub, OpenAI, Mediastack, Google Apps Script).

Composants Clés :

  • **Frontend :** Pages HTML, CSS et JavaScript client-side (incluant des scripts pour le chargement dynamique du header, la gestion des filtres, l'interaction utilisateur).
  • **Backend (Serverless) :** Netlify Functions (fichiers .js dans le dossier netlify/functions/) pour gérer la logique côté serveur (upload d'images, persistance des prompts/profils, requêtes API externes).
  • **Stockage de Données :**
    • **Côté client (localStorage) :** Utilisé pour la persistance locale et temporaire des demandes IA, tips et profils d'équipe.
    • **Côté serveur (via Netlify Functions) :** Principalement des fichiers JSON (prompts.json, profil.json) stockés dans un dépôt GitHub, et potentiellement Google Sheets via Google Apps Script.
  • **Déploiement :** Netlify.

3. Gestion des Liens et Navigation

La navigation sur le site s'effectue par une combinaison de liens HTML classiques et de redirections JavaScript.

3.1. Liens HTML Statiques

  • Les liens sont définis à l'aide de balises <a> avec des attributs href pointant vers des chemins relatifs (ex: index.html, outils.html, equipe.html).
  • La barre de navigation principale est contenue dans header.html et est incluse dynamiquement dans presque toutes les pages pour assurer la cohérence.

3.2. Inclusion Dynamique du Header

  • **Mécanisme :** Presque toutes les pages HTML (e.g., index.html, outils.html, cas-usages.html, prompts.html, profil.html, veille.html, equipe.html, charte.html, faq.html, demandes_ia.html, compresseur.html, creer-tip.html) utilisent le code JavaScript suivant pour charger et insérer le header :
    • fetch("header.html")
  .then(res => res.text())
  .then(html => {
    document.getElementById("header-placeholder").innerHTML = html;
    // ... (autres initialisations comme mettreAJourBulleDemandes et liens actifs)
  })
  .catch(error => {
    console.error("Erreur de chargement du header:", error);
  });
  • **Avantage :** Facilite la maintenance et l'uniformisation du menu de navigation.

3.3. Gestion des Liens Actifs

  • Un script JavaScript commun (présent dans plusieurs pages) identifie la page actuelle via location.pathname et ajoute la classe active au lien correspondant dans la navigation, offrant une indication visuelle :
    • const path = location.pathname.split("/").pop();
const links = document.querySelectorAll("nav a");
links.forEach(link => {
  if (link.getAttribute("href") === path) {
    link.classList.add("active");
  }
});

3.4. Redirections JavaScript

  • **Après Soumission de Prompt (prompts.html) :** Redirection vers galerie.html après un délai de 3 secondes : window.location.href = "/galerie.html";
  • **Édition de Profil (fiche.html vers profil.html) :** Passage de paramètres via l'URL pour indiquer le profil à éditer : window.location.href = "profil.html?edit=" + encodeURIComponent(nomRecherche);
  • **Après Sauvegarde de Profil (profil.html) :** Redirection vers equipe.html une fois le profil enregistré et le déploiement vérifié : window.location.href = "equipe.html";

4. Gestion des Données

La gestion des données est répartie entre le stockage local du navigateur et des interactions avec des services tiers via des Netlify Functions.

4.1. Stockage Local du Navigateur (localStorage)

Le localStorage est utilisé pour des données non sensibles et persistantes localement pour l'utilisateur. Ces données sont spécifiques au navigateur et ne sont pas synchronisées côté serveur.

  • **Demandes IA :** Les données des demandes sont stockées sous la clé "demandesIA". Exemple d'accès : JSON.parse(localStorage.getItem("demandesIA") || "[]").
  • **Tips :** Les "tips" créés sont stockés sous la clé "tips". Exemple d'accès : localStorage.getItem('tips').
  • **Profils d'Équipe :** Les profils d'équipe sont stockés sous la clé "profilsEquipe". Exemple d'accès : localStorage.getItem("profilsEquipe"). Il est important de noter que ces profils sont également gérés côté serveur via GitHub.

4.2. Persistance des Données via Netlify Functions (Backend)

Les Netlify Functions agissent comme une couche backend serverless, gérant la logique métier, l'authentification et les interactions avec les APIs externes.

4.2.1. Stockage GitHub (Base de Données de Fichiers JSON)

Le principal mécanisme de persistance pour les prompts et les profils est un dépôt GitHub spécifique : spiresm/IA_marketing. Les Netlify Functions interagissent avec l'API GitHub pour lire, écrire et mettre à jour des fichiers JSON.

  • **netlify/functions/getPrompts.js :** Lit prompts.json. Authentification via process.env.GITHUB_TOKEN.
  • **netlify/functions/pushPrompt.js :** (Probablement appelée par prompts.html ou BK3_prompts.html) Ajoute/met à jour des prompts dans prompts.json. Authentification via process.env.GITHUB_TOKEN.
  • **netlify/functions/getProfils.js :** Récupère les profils depuis netlify/functions/profil.json. Authentification via process.env.GITHUB_TOKEN.
  • **netlify/functions/updateProfil.js :** (Appelée par profil.html) Crée/met à jour un profil dans netlify/functions/profil.json. Authentification via process.env.GITHUB_TOKEN.
  • **netlify/functions/deleteProfil.js :** Supprime un profil de netlify/functions/profil.json. Authentification via process.env.GITHUB_TOKEN.
  • **netlify/functions/BK_uploadImage.js (ou uploadImage) :** Gère l'upload d'images vers le dossier images/ du dépôt GitHub. Authentification via process.env.GITHUB_TOKEN.

    Note importante : En local, Netlify ne parse pas automatiquement les corps de requête multipart/form-data. L'upload d'images pourrait ne fonctionner qu'une fois déployé sur Netlify, ou nécessiter une configuration spécifique pour le développement local.

4.2.2. Intégrations API Externes (Clés API / Tokens)

Les fonctions suivantes interagissent avec des services tiers, nécessitant des clés API définies comme variables d'environnement sur Netlify.

  • **netlify/functions/chat-clean.js :** Interface avec l'API OpenAI pour le chatbot. Utilise process.env.OPENAI_API_KEY et le modèle gpt-3.5-turbo.
  • **netlify/functions/fetchNews.js :** Récupère des articles d'actualité via l'API Mediastack. Utilise process.env.MEDIASTACK_API_KEY.
  • **netlify/functions/gas-proxy.js :** Agit comme un proxy pour les requêtes vers une Google Apps Script (GAS). L'URL GAS est incluse directement dans la fonction. Cela permet une communication sécurisée avec des feuilles Google Sheets ou d'autres services Google.
  • **netlify/functions/sendrequest.js :** Gère l'envoi des "demandes" (probablement via un service d'email et/ou une Google Sheet via la GAS).

    Attention : Le code mentionne l'envoi d'emails mais ne détaille pas une intégration directe avec un service (ex: SendGrid). Cela pourrait nécessiter une configuration supplémentaire ou l'appel d'une autre Netlify Function dédiée à l'envoi d'emails.

  • **netlify/functions/saveDemande.js :** Sauvegarde les demandes soumises.

    Problème de Persistance : Cette fonction sauvegarde les données dans /tmp/demandes.json. **Le répertoire /tmp sur Netlify est éphémère et non persistant.** Les données sauvegardées ici seront perdues entre les invocations de fonctions ou les déploiements. Il est critique de rediriger ces données vers un stockage persistant (ex: GitHub via une autre fonction, une base de données, ou via sendrequest.js vers Google Sheets) si elles doivent être conservées.

4.2.3. Monitoring de Déploiement

  • **netlify/functions/checkDeployStatus.js :** Vérifie le statut du dernier déploiement Netlify en interrogeant l'API Netlify. Utilise process.env.NETLIFY_SITE_ID et process.env.NETLIFY_API_TOKEN. Cette fonction est appelée par profil.html pour s'assurer que les modifications de profil sont bien en ligne avant de rediriger l'utilisateur.

5. Déploiement

Le projet est déployé sur **Netlify**, une plateforme d'hébergement et de CI/CD (intégration et livraison continues) pour les applications web statiques et les fonctions serverless.

5.1. Caractéristiques de Déploiement Netlify

  • **Intégration Git :** Le site est connecté à un dépôt GitHub (`spiresm/IA_marketing`). Chaque push sur la branche `main` (ou une autre branche configurée) déclenche un nouveau déploiement automatique.
  • **Netlify Functions :** Les fichiers JavaScript dans le dossier `netlify/functions/` (ou spécifiés dans `netlify.toml` s'il existe) sont automatiquement déployés comme des fonctions serverless et accessibles via `/.netlify/functions/NOM_DE_LA_FONCTION`.
  • **Variables d'Environnement :** Toutes les clés API (`GITHUB_TOKEN`, `OPENAI_API_KEY`, `NETLIFY_API_TOKEN`, `MEDIASTACK_API_KEY`, etc.) doivent être configurées dans le tableau de bord Netlify sous "Site settings" > "Build & deploy" > "Environment variables". C'est crucial pour la sécurité et le bon fonctionnement des fonctions serverless.
  • **Déploiement Atomique :** Netlify déploie de nouvelles versions de manière atomique, ce qui signifie que les changements ne sont visibles qu'une fois que l'ensemble du déploiement est terminé et prêt.
  • **Rollbacks Faciles :** Netlify permet de revenir facilement à une version précédente du site en cas de problème.

5.2. Étapes de Déploiement (Générales)

  1. **Code :** Le code source (HTML, CSS, JS frontend, Netlify Functions) est versionné dans un dépôt GitHub (ex: spiresm/IA_marketing).
  2. **Configuration Netlify :**
    • Le site est connecté au dépôt GitHub sur la plateforme Netlify.
    • Les commandes de build (ex: npm run build si un framework frontend est utilisé, ou simplement la publication des fichiers statiques) sont configurées.
    • Les variables d'environnement nécessaires pour les Netlify Functions sont définies.
  3. **CI/CD :** Chaque push vers la branche configurée (généralement main) déclenche un nouveau build et déploiement sur Netlify.
  4. **Fonctionnement :**
    • Les fichiers HTML, CSS, et JavaScript frontend sont servis directement par le CDN de Netlify.
    • Les requêtes vers /.netlify/functions/* sont acheminées vers les Netlify Functions correspondantes qui s'exécutent dans un environnement serverless.

6. Recommandations et Points d'Attention

  • **Persistance des demandes.json :** Le stockage dans /tmp pour saveDemande.js n'est pas persistant. Si ces données doivent être conservées, il est impératif de les écrire dans GitHub (comme les profils et prompts) ou dans une base de données réelle (ex: FaunaDB, MongoDB Atlas) ou un service comme Google Sheets (via Google Apps Script).
  • **Sécurité des Clés API :** Assurez-vous que toutes les clés API sont uniquement stockées en tant que variables d'environnement sur Netlify et ne sont jamais directement exposées dans le code côté client ou les fichiers de configuration du dépôt.
  • **Gestion des Erreurs :** Les fonctions Netlify intègrent déjà une bonne gestion des erreurs, mais assurez-vous que le frontend affiche des messages compréhensibles à l'utilisateur en cas d'échec des appels API.
  • **Gestion des Branches :** Pour le développement, utilisez des branches de fonctionnalité et fusionnez dans main uniquement lorsque le code est stable pour éviter de casser le site en production. Netlify permet des "deploy previews" pour chaque pull request.
  • **Performance :** Continuez à optimiser les images et les assets, et minimisez le code JavaScript pour améliorer les temps de chargement.
  • **Évolutivité du Stockage GitHub :** Utiliser GitHub comme base de données de fichiers JSON est simple pour des petits volumes de données. Pour des données plus structurées, des volumes importants ou des requêtes complexes, envisagez une base de données NoSQL ou SQL dédiée.
